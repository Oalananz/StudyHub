# 🚀 Deploying Study Hub

Two supported targets. The whole stack is configured for **Render** (one platform, free tier).
You can also mix hosts (e.g. Vercel/Netlify frontend + Supabase DB) — see the bottom.

---

## Option A — Everything on Render (recommended)

Render runs all three pieces: the ASP.NET API (Docker), PostgreSQL, and the Next.js
frontend (static). It's defined as a Blueprint in [`render.yaml`](render.yaml).

### Steps

1. **Push this repo to GitHub** (Render deploys from a Git repo).
   ```bash
   git init && git add . && git commit -m "Study Hub"
   git branch -M main
   git remote add origin https://github.com/<you>/study-hub.git
   git push -u origin main
   ```

2. **Create the Blueprint** on Render:
   - Go to <https://dashboard.render.com> → **New** → **Blueprint**.
   - Connect your GitHub repo. Render reads `render.yaml` and proposes 3 resources:
     `studyhub-db`, `studyhub-api`, `studyhub-web`.
   - Click **Apply**. The database and API deploy automatically.

3. **Wire the two URLs** (one-time, because each needs the other's address):
   - Open **studyhub-web** → **Environment** → set
     `NEXT_PUBLIC_API_URL = https://studyhub-api.onrender.com`
     (use the real API URL shown on its service page).
   - Open **studyhub-api** → **Environment** → set
     `Cors__AllowedOrigins__0 = https://studyhub-web.onrender.com`
     (use the real frontend URL).
   - **Manual Deploy → Deploy latest commit** on **studyhub-web** so the API URL is baked
     into the static build. (CORS on the API takes effect without a rebuild.)

4. Open the frontend URL and register an account. Done. 🎉

### Notes & gotchas
- **Cold starts:** free web services sleep after ~15 min idle; the first request then
  takes ~30–60 s while the API wakes. The static frontend never sleeps.
- **Free Postgres expires after 30 days.** Before then, either upgrade the DB, or switch to
  a Supabase database (below) — the API already supports both formats.
- The API auto-creates its schema on first boot (`EnsureCreated`) and binds to Render's
  `$PORT`. `Jwt__Key` is generated by Render automatically.

## Option B — Render API + Supabase database (longer-lived free DB)

Use Supabase for a free Postgres that doesn't expire, keep the API/frontend on Render.

1. Create a project at <https://supabase.com>.
2. **Project Settings → Database → Connection string → URI.** Copy it; it looks like:
   ```
   postgresql://postgres:[PASSWORD]@db.xxxx.supabase.co:5432/postgres
   ```
3. On Render, open **studyhub-api** → **Environment**, and **replace**
   `ConnectionStrings__Default` with that URI (remove the `fromDatabase` link / `studyhub-db`).
4. Redeploy the API. It converts the `postgresql://` URL to the Npgsql format automatically
   (see `NormalizePostgresConnectionString` in `Program.cs`) and creates the tables on boot.

> The same URI works locally too: set the `ConnectionStrings__Default` env var before
> `dotnet run`.

## Option C — Frontend on Vercel / Netlify / GitHub Pages

The frontend is a static export, so it can live anywhere that serves static files. The API
still needs a .NET-capable host (Render/Railway/Fly.io) — Vercel, Netlify and GitHub Pages
**cannot** run the ASP.NET server.

- **Build command:** `npm run build`  ·  **Output dir:** `out`
- **Required build env:** `NEXT_OUTPUT=export` and `NEXT_PUBLIC_API_URL=<your API URL>`
- **GitHub Pages only:** because the site is served under `/<repo>/`, also set
  `basePath`/`assetPrefix` in `next.config.js` to `/<repo>` — not needed on Vercel/Netlify/Render.

Whatever host you pick, remember to add that origin to the API's
`Cors__AllowedOrigins__0`.
